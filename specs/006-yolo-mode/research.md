# Research: YOLO Mode

**Feature**: 006-yolo-mode | **Date**: 2026-03-14

## R1: Claude `--dangerously-skip-permissions` + Deny Rule Interaction

**Decision**: Deny rules override `--dangerously-skip-permissions`. The evaluation order is: deny → ask → allow, with bypass mode only affecting the final fallback.

**Rationale**: Official Claude Code docs confirm deny rules are evaluated before the permission mode check. A tool denied at any level cannot be allowed by any other level. This means `settings.yolo.json` with only `permissions.deny` entries works correctly with `--dangerously-skip-permissions`.

**Alternatives considered**:
- Hooks-based blocking: More reliable (OS-level) but separate from twsrt's scope
- Sandbox-only restrictions: Already generated by twsrt independently of yolo mode

**Caveat**: GitHub issue #8961 reports deny rules in `.claude/settings.local.json` are sometimes ignored. Sandbox restrictions (also generated by twsrt) provide a backstop.

## R2: Copilot `--yolo` Flag + `--deny-tool` Interaction

**Decision**: Copilot's `--yolo` (alias for `--allow-all`) is overridden by `--deny-tool`. Deny always wins.

**Rationale**: Official GitHub docs explicitly state: "Deny rules always take precedence over allow rules, even when --allow-all is set." The `--deny-tool` option takes precedence over `--allow-all-tools` and `--allow-tool`.

**Alternatives considered**: None — the behavior is explicitly documented and unambiguous.

## R3: Allow Statement Handling in YOLO Mode

### Claude: `permissions.allow`

**Decision**: Include `permissions.allow` with `WebFetch(domain:...)` entries in yolo mode output.

**Rationale**:
- In standard mode, ClaudeGenerator produces `allow` entries only from `NETWORK/ALLOW` rules (WebFetch domain entries). No blanket tool allows (Read, Edit, etc.) are generated — those are preserved via `selective_merge()` from the user's existing settings.json.
- In yolo mode, `settings.yolo.json` is a standalone file (no selective_merge). The WebFetch domain entries should still be present because:
  1. They correspond to `sandbox.network.allowedDomains` — the permission and sandbox layers should be consistent
  2. They are harmless in bypass mode (redundant but not conflicting)
  3. If someone uses `settings.yolo.json` without bypass mode, the allow entries make the config functional
- `permissions.ask` is omitted entirely (core yolo behavior)

### Claude: `permissions.deny`

**Decision**: Include only deny entries derived from bash-rules deny section. SRT-derived deny entries (READ/DENY, WRITE/DENY) are still included since they come from SRT, not bash-rules.

**Rationale**: FR-002 says "exclude all rules derived from the 'ask' section of bash-rules.json." FR-006 says "all non-bash-rules configuration MUST be included identically." SRT filesystem deny rules are non-bash-rules and must remain.

### Copilot: `--allow-tool` and `--allow-url` entries

**Decision**: Omit `--allow-tool` and `--allow-url` entries in yolo mode. They are redundant since `--yolo` subsumes `--allow-all-tools`, `--allow-all-paths`, and `--allow-all-urls`.

**Rationale**:
- `--yolo` expands to `--allow-all`, which includes all three allow-all flags
- Emitting redundant `--allow-tool 'shell(*)'` or `--allow-url 'example.com'` after `--yolo` adds noise without functional value
- `--deny-tool` and `--deny-url` entries ARE still needed (deny overrides allow)

### Copilot: `--deny-url` entries

**Decision**: Include `--deny-url` entries from SRT network denied domains in yolo mode.

**Rationale**: FR-006 requires non-bash-rules config to be included identically. `--deny-url` entries override `--yolo`'s `--allow-all-urls`, providing the correct deny-overrides-allow behavior.

## R4: YOLO Mode Implementation Approach

**Decision**: Add `yolo: bool = False` field to `AppConfig` dataclass. Each generator checks `config.yolo` internally.

**Rationale**:
- No Protocol change needed — `AppConfig` is already passed to `generate()` and `diff()`
- Each agent has different yolo semantics (Claude: omit ask + keep allow; Copilot: prepend `--yolo` + omit `--allow-*` + omit ask-derived deny-tool)
- Filtering at CLI level alone is insufficient: Copilot needs to add `--yolo` flag and omit `--allow-*` entries, which are structural changes to the output

**Alternatives considered**:
- Filter rules at CLI level only: Simpler but can't handle Copilot's `--yolo` flag insertion or `--allow-*` omission
- Add `yolo` parameter to Protocol methods: Breaks Protocol, requires updating all implementations and tests. AppConfig already serves as the config carrier.

## R5: YOLO Output Path Derivation

**Decision**: Helper function `yolo_path(original: Path) -> Path` inserts `.yolo` before the file extension. Configurable via optional `config.toml` keys.

**Rationale**: Consistent, predictable naming. `settings.json` → `settings.yolo.json`, `copilot-flags.txt` → `copilot-flags.yolo.txt`.

**Config override**: Add optional keys `claude_settings_yolo` and `copilot_output_yolo` to `[targets]` section of config.toml. If absent, derive automatically.
